## 2. Frozen Zone и версии плана: формализация

### 2.1. Новая сущность `PlanVersion`

**Назначение:** Версия стратегического/тактического плана, результат конкретного пересчёта.

**Поля (концептуально):**
- `id` (UUID) — уникальный идентификатор версии
- `name` (string) — название версии (например, "Операционный план 2026‑02‑03 1‑я смена")
- `created_at` (datetime) — дата/время создания
- `created_by` (UUID, nullable) — пользователь, создавший версию
- `status` (enum) — статус версии:
  - `DRAFT` — черновик, не влияет на производство
  - `ACTIVE` — активная версия, используется для планирования
  - `ARCHIVED` — архивная версия (предыдущая активная)
  - `REJECTED` — отклонённая версия
- `horizon_start` (date) — начало горизонта планирования
- `horizon_end` (date) — конец горизонта планирования
- `frozen_until` (datetime) — граница frozen zone (дата/время, до которой план заморожен)
- `scenario_type` (enum) — тип сценария:
  - `PRODUCTION` — производственный сценарий (может быть активным)
  - `WHAT_IF` — сценарий "что‑если" (только для анализа)
- `input_parameters` (JSON) — входные параметры расчёта:
  - `orders_filter` — фильтры по заказам
  - `horizon_days` — горизонт планирования
  - `capacity_profile` — профиль мощностей
  - и т.д.

**Связи:**
- `ManufacturingOrder.plan_version_id` (nullable) — для внутренних/производственных заказов, созданных MES в рамках этой версии
- `Batch.plan_version_id` (nullable) — для партий, созданных в рамках этой версии
- `ProductionTask.plan_version_id` (nullable) — для задач, созданных в рамках этой версии

**Правило:** В системе в каждый момент времени **ровно одна** `PlanVersion` со статусом `ACTIVE` и `scenario_type=PRODUCTION`.

---

### 2.2. Изменения в существующих сущностях

#### 2.2.1. `ManufacturingOrder`

**Добавить поля:**
- `plan_version_id` (UUID, nullable, FK → `plan_versions.id`) — для внутренних/производственных заказов, созданных MES
- `planning_status` (enum) — статус планирования:
  - `UNPLANNED` — не включён в план
  - `PLANNED` — включён в план, но не утверждён
  - `RELEASED` — утверждён и выпущен в производство
  - `IN_PROGRESS` — выполняется
  - `COMPLETED` — завершён
  - `CANCELLED` — отменён

**Семантика frozen zone для заказов:**
- Для клиентских (из 1С) заказов мы **не переписываем** зафиксированные статусы/разбиения, даже если 1С изменила due_date, пока соответствующие задачи/партии находятся в frozen zone
- Конфликт показываем на вкладке "Стратегия" как предупреждение

---

#### 2.2.2. `Batch`

**Добавить поля:**
- `plan_version_id` (UUID, nullable, FK → `plan_versions.id`)
- `planning_state` (enum) — состояние планирования:
  - `DRAFT` — черновик (создан алгоритмом, но не утверждён)
  - `APPROVED` — утверждён (включён в активный план)
  - `FROZEN` — заморожен (находится в frozen zone)
  - `EXECUTED` — выполнен
  - `CANCELLED` — отменён
- `frozen_from` (datetime, nullable) — дата/время, когда партия попала в frozen zone

**Frozen logic:**
- Batch со `planning_state=FROZEN` нельзя:
  - автоматом перемещать/удалять при новых пересчётах
  - менять объём и дату начала, кроме как через отдельный "ручной override" (подтверждённый пользователем)

---

#### 2.2.3. `ProductionTask`

**Добавить поля:**
- `plan_version_id` (UUID, nullable, FK → `plan_versions.id`)
- `planning_state` (enum) — состояние планирования:
  - `DRAFT` — черновик
  - `APPROVED` — утверждён
  - `FROZEN` — заморожен
  - `EXECUTED` — выполнен
  - `CANCELLED` — отменён
- `is_frozen` (boolean, computed) — вычисляемое поле: `true`, если `scheduled_start < frozen_until` активной версии

**Правила:**
- Планировщик может редактировать `ProductionTask` в frozen zone **только вручную** (drag‑and‑drop в Гантте с подтверждением)
- Планировщик‑алгоритм при новых расчётах:
  - читает все задачи из активной версии, оставляет **FROZEN** как есть
  - пересчитывает только задачи с `planning_state` ∈ {`DRAFT`, `APPROVED`} и `scheduled_start >= frozen_until`

---

### 2.3. Формальное определение Frozen Zone

**Определение:**
- Пусть:
  - `now` — текущее время
  - `frozen_horizon_days` — настраиваемый параметр (например, 2‑3 смены / 1‑2 дня)
  - `frozen_until = floor_to_day(now) + frozen_horizon_days`
- **Frozen zone** — временной интервал [now; frozen_until)
- Все `ProductionTask` и `Batch`, у которых `scheduled_start < frozen_until`, считаются **замороженными**

**В `PlanVersion`:**
- Значение `frozen_until` зашито как параметр версии
- При принятии нового плана frozen zone может "переезжать" вперёд во времени (rolling horizon)

---

### 2.4. Жизненный цикл версии плана

#### 2.4.1. Создание DRAFT версии
- Пользователь на вкладке "Стратегия" выбирает фильтры, запускает `Рассчитать план`
- Создаётся `PlanVersion` со статусом `DRAFT`
- Алгоритм:
  - копирует FROZEN‑задачи/партии из текущей `ACTIVE` версии
  - пересчитывает остальные задачи/партии на интервале [frozen_until; horizon_end]

#### 2.4.2. Анализ DRAFT
- Пользователь просматривает MRP‑таблицу, Гантт, загрузку ресурсов
- Может откатить и изменить фильтры/приоритеты → новая `DRAFT` (предыдущую можно пометить `REJECTED`)

#### 2.4.3. Принятие плана
- Нажатие `Принять план`:
  - текущая `ACTIVE` версия становится `ARCHIVED`
  - `DRAFT` → `ACTIVE` (для `scenario_type=PRODUCTION`)
  - все Batch/ProductionTask ссылаются на новую `plan_version_id` (либо переносятся по ключу)

#### 2.4.4. Выполнение
- По мере фактизации:
  - `ProductionTask.planning_state` меняется на `EXECUTED`
  - `Batch` — `COMPLETED`
  - `ManufacturingOrder` — `IN_PROGRESS` / `COMPLETED`
- Эти статусы учитываются при следующих пересчётах (такие задачи не пересчитываются)

---

### 2.5. Поведение при новых данных из 1С

**Когда приходит новая порция данных из 1С:**

1. **Сервис сравнения заказов** (уже есть) даёт:
   - новые, изменённые, удалённые заказы

2. **На вкладке "Стратегия":**
   - показываем счётчики и списки по этим изменениям (как сейчас на странице "Изменения заказов")
   - пользователь решает:
     - какие заказы **учитывать** в новом сценарии (MRP / позаказный вид)
     - какие пока проигнорировать или отложить

3. **При запуске нового пересчёта:**
   - frozen zone (из текущей `ACTIVE` версии) остаётся неизменной
   - новые/изменённые заказы влияют только на интервал [frozen_until; horizon_end]
   - если изменение 1С "конфликтует" с уже замороженным участком (например, заказ отменён, а задачи уже в frozen zone), это помечаем как **конфликт плана**:
     - визуально на вкладке "Стратегия" и/или в Гантте
     - решается только ручным вмешательством (отмена задач / перенос заморозки)

## 3. Технические детали реализации

### 3.1. Миграции БД

**Новая таблица `plan_versions`:**
```sql
CREATE TABLE plan_versions (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    name VARCHAR(255) NOT NULL,
    created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
    created_by UUID REFERENCES users(id),
    status VARCHAR(20) NOT NULL CHECK (status IN ('DRAFT', 'ACTIVE', 'ARCHIVED', 'REJECTED')),
    horizon_start DATE NOT NULL,
    horizon_end DATE NOT NULL,
    frozen_until TIMESTAMP WITH TIME ZONE NOT NULL,
    scenario_type VARCHAR(20) NOT NULL CHECK (scenario_type IN ('PRODUCTION', 'WHAT_IF')),
    input_parameters JSONB,
    CONSTRAINT unique_active_production UNIQUE (status, scenario_type) WHERE status = 'ACTIVE' AND scenario_type = 'PRODUCTION'
);
```

**Изменения в существующих таблицах:**
- `manufacturing_orders`: добавить `plan_version_id UUID REFERENCES plan_versions(id)`, `planning_status VARCHAR(20)`
- `batches`: добавить `plan_version_id UUID REFERENCES plan_versions(id)`, `planning_state VARCHAR(20)`, `frozen_from TIMESTAMP WITH TIME ZONE`
- `production_tasks`: добавить `plan_version_id UUID REFERENCES plan_versions(id)`, `planning_state VARCHAR(20)`

### 3.2. Backend сервисы

### 3.3. Frontend компоненты

**Новые компоненты:**
