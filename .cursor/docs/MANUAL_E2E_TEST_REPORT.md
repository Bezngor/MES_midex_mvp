# Отчёт о ручном сквозном тестировании MES (этап 0.0)

**Дата завершения этапа:** январь 2026  
**План тестирования:** `.cursor/docs/MANUAL_TEST_PLAN_MES.md`  
**Решение:** текущую фазу ручного тестирования функционала приложения **завершить**. Полноценное точечное тестирование жизненного цикла заказа (стратегическое планирование → создание → тактическое планирование → выпуск в производство) невозможно из‑за отсутствия UI для заказов и незавершённости UI DSIZ.

---

## 1. Проведённые проверки и результаты

### 1.1 Продукты (раздел 1)

- Страница «Продукты» открывается, список загружается.
- Создание, редактирование, удаление продукта, фильтр по типу — проверялись в рамках использования системы; детальный чек-лист по каждому пункту не фиксировался в данном отчёте.

### 1.2 BOM (раздел 2)

- Страница «BOM» открывается, выбор ГП/Масса, просмотр и редактирование спецификации — доступны.
- Исправлена точность количества в BOM (миграция `bom_quantity_precision`), скорректирована валидация схемы.

### 1.3 Партии (раздел 3)

- Страница «Партии» — список батчей, фильтр по статусу, создание батча, «Запустить» / «Завершить» — функционал присутствует.

### 1.4 Остатки (раздел 4)

- Страница «Остатки» — таблица остатков, фильтр по типу остатка.
- Добавлены экспорт списка продуктов для заполнения остатков, конвертер xlsx→CSV, загрузчик остатков из CSV. Статус отображения изменён: «Готовая продукция» → «Готово к отпуску» (единая формулировка для сырья, массы, ГП).

### 1.5 Расписание (раздел 5)

- Страница «Расписание» — загрузка Ганта, выбор диапазона (7/14/30 дней), блоки выпуска заказов и выпущенных заказов, проверка маршрутов и правил РЦ.

### 1.6 MRP (раздел 6)

- **6.1–6.3** Консолидация заказов: таблица с колонками по приоритетам (URGENT, HIGH, NORMAL, LOW), переключатели видимости колонок, «ИТОГО» и «Крайний срок» по выбранным приоритетам, горизонт планирования.
- **6.4** Взрыв BOM: в UI — модальное окно по клику на продукт; «Количество ГП» в модалке считается по сумме **выбранных** приоритетных колонок (не по общей сумме). API `POST /api/v1/mrp/explode-bom` — проверен.
- **6.5** Нетто-потребность: API `POST /api/v1/mrp/net-requirement` — проверен, расчёт с учётом резервов корректен.
- **6.6** Создание заказа на варку: API `POST /api/v1/mrp/create-bulk-order` — проверен (округление до батча, тип INTERNAL_BULK). Пункт с `parent_order_id` отложен до появления списка заказов.
- **6.7** Граничные случаи — частично; полноценная проверка ограничена отсутствием UI заказов.

### 1.7 DSIZ Planning (раздел 7)

- Вкладка открывается, форма параметров (дата, горизонт, слоты, персонал, режим маркировки), кнопка «Запустить планирование».
- Добавлено ограничение: дата начала планирования не раньше сегодня (frontend + backend).
- Исправлено отображение результатов: ответ API (плоский объект без обёртки `data`) обрабатывается в хуке, Gantt и сводка отображаются.
- **Итог:** модель отображения Gantt не соответствует данным (12 рядов «реакторов» при 2 сменах × 2 слотах), блоки смещены относительно разбивки по дням, алгоритм размещения операций неочевиден. **Вся вкладка перенесена в бэклог** после Dispatching.

### 1.8 DSIZ Shift Actualize (раздел 8)

- Вкладка «Фактизация смены»: дата смены, номер смены, поле «ID заказов (через запятую)» — только ручной ввод; кнопка «Запустить диспетчеризацию» вызывает preview диспетчеризации (таблица задач, конфликты). Кнопки «Сохранить факт» и «Напечатать задание смены» — заглушки. Дата и номер смены в API не передаются.
- **Итог:** без списка заказов из системы точечное тестирование цепочки невозможно; вкладка в зачаточном состоянии.

### 1.9 DSIZ Master Data (раздел 9)

- Вкладка открывается; по оценке пользователя — заглушка на всё до следующих итераций.

### 1.10 Общее (раздел 10)

- Навигация между разделами работает, критичных падений при переходах не зафиксировано.

## 2. Ограничения, выявленные в ходе тестирования

1. **Нет UI для заказов:** создания и просмотра производственных заказов (ManufacturingOrder) в интерфейсе нет. Компоненты `OrderForm` и `OrderList` есть в коде, но ни на одной странице не используются. Заказы создаются только через API или загрузку CSV.
2. **Нет сквозного жизненного цикла в UI:** невозможно пройти в приложении цепочку «стратегическое планирование → создание заказа → тактическое планирование → выпуск в производство» — нет ни вкладки заказов, ни связки DSIZ Planning / Shift Actualize с реальными заказами и сменами.
3. **DSIZ Planning:** Gantt и модель данных (смены/слоты/дни) требуют переработки; вкладка целиком вынесена в бэклог.
4. **DSIZ Shift Actualize:** ввод заказов только вручную (UUID через запятую); дата/смена не участвуют в API; заглушки на сохранение факта и печать задания.
5. **DSIZ Master Data:** функционал в виде заглушек до следующих итераций.

## 3. Задачи бэклога

Ниже — задачи, перенесённые в бэклог по результатам тестирования и обсуждения. Они должны войти в план Phase 4 в рамках создания UI стратегического и тактического планирования (см. раздел 4).

### 3.1 DSIZ Planning (вкладка целиком)

- **7.бэклог** Полная доработка и тестирование вкладки DSIZ Planning:
  - приведение модели отображения в соответствие с данными (смены, слоты, дни; не 12 абстрактных «реакторов»);
  - исправление позиционирования блоков Gantt относительно разбивки по дням;
  - учёт приоритета MRP (URGENT → HIGH → NORMAL → LOW) при размещении задач в плане;
  - определение смысла и использование «Состав персонала по сменам» (стратегия/тактика);
  - определение места «Режим маркировки» (тактическое планирование);
  - понятная визуализация плана и, при необходимости, сохранение плана в БД.

### 3.2 DSIZ Shift Actualize

- Выбор заказов для смены не вручную (UUID), а из списка заказов с фильтром по дате/смене/статусу.
- Передача даты и номера смены в API диспетчеризации (или в отдельный API фактизации).
- Реализация «Сохранить факт» и «Напечатать задание смены» (или явные заглушки с описанием контракта).

### 3.3 DSIZ Master Data

- Реализация просмотра и при необходимости редактирования справочников DSIZ (режимы РЦ, матрица переналадок, маршруты продуктов, базовые ставки, потребность в персонале) вместо заглушек.

### 3.4 Заказы и жизненный цикл

- Включение в навигацию вкладки «Заказы» (или аналога) с использованием существующих `OrderForm` / `OrderList` или их доработка.
- Обеспечение возможности создания и просмотра производственных заказов в UI, фильтрации по статусу/дате и связи с MRP, расписанием и DSIZ Shift Actualize.

## 4. Рекомендации по плану Phase 4

**Обязательно включить в тактический план Phase 4** отдельный блок (ключевой алгоритм MES для предприятия):

**«UI для реализации концепций Стратегического и Тактического планирования»**

В этот блок входят:

- все задачи бэклога по вкладкам **DSIZ Planning**, **DSIZ Shift Actualize**, **DSIZ Master Data** (п. 3.1–3.3);
- задачи по **UI заказов** и сквозному жизненному циклу заказа (п. 3.4).

**Причины выделения в отдельный блок:**

1. **Ключевой алгоритм MES:** для данного предприятия цепочка «стратегическое планирование (горизонт, нетто-потребность, варки) → тактическое планирование (смена, заказы, диспетчеризация) → выпуск в производство и фактизация» является ядром MES. Без полноценного UI эта цепочка не может быть ни протестирована, ни использована в эксплуатации.
2. **Разделение стратегии и тактики:** выявлено смешение в текущем UI (параметры на период vs параметры на смену/день). Выделение блока закрепляет необходимость явного разделения и доработки обоих уровней.
3. **Зависимость остальных доработок:** расширенная диспетчеризация, автоматическое создание партий по плану, drag‑and‑drop в Gantt и т.д. опираются на рабочий UI планирования и фактизации; без него они не дают законченного сценария для пользователя.

**Статус в плане:** ключевой алгоритм MES для данного предприятия; выполнение блока обязательно для перехода к production-ready сценариям планирования и исполнения смен.
