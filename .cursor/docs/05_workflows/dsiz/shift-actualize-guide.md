# DSIZ Shift Actualize — описание вкладки

**Путь:** `/dsiz/shift-actualize`  
**Заголовок в UI:** «Фактизация смены»

---

## Назначение

Вкладка предназначена для **тактической работы со сменой**: выбор смены (дата + номер), ввод/выбор заказов для диспетчеризации, запуск расчёта расписания задач по рабочим центрам и (в перспективе) сохранение фактических данных и печать задания смены.

Сейчас основное действие — **preview диспетчеризации**: по списку ID производственных заказов система считает, как задачи разложатся по РЦ, с учётом приоритетов, переналадки и выявляет конфликты. В БД задачи при этом **не назначаются** — это только превью.

---

## Элементы интерфейса

| Элемент | Описание |
|--------|----------|
| **Дата смены** | Дата, на которую смотрите смену (календарь). |
| **Номер смены** | Смена 1 или Смена 2. |
| **ID заказов (через запятую)** | Список UUID производственных заказов для расчёта диспетчеризации. Заказы должны быть в статусе, допускающем диспетчеризацию (например, RELEASED с задачами QUEUED). |
| **Запустить диспетчеризацию** | Вызов `POST /api/v1/dsiz/dispatching/run` с переданными `manufacturing_order_ids`. Возвращает превью: список задач с началом/окончанием, РЦ, длительностью, переналадкой и список конфликтов (пересечения на одном РЦ). |
| **Сохранить факт** | Заглушка: «Сохранение фактических данных будет реализовано в следующей версии». |
| **Напечатать задание смены** | Заглушка: «Печать задания смены будет реализована в следующей версии». |

**Важно:** Дата и номер смены в текущей реализации **не передаются** в API диспетчеризации — в запрос уходят только `manufacturing_order_ids`. То есть выбор смены пока только для контекста пользователя; логика «на эту смену» в бэкенде не завязана.

---

## API: POST /api/v1/dsiz/dispatching/run

**Тело запроса:**
- `manufacturing_order_ids` — массив UUID заказов (обязательно).
- `work_center_id` — UUID РЦ (опционально), фильтр превью по одному РЦ.

**Ответ 200:**
- `success: true`
- `data.gantt_preview` — массив задач: task_id, order_id, work_center_id, work_center_name, task_start, task_end, duration_hours, changeover_minutes, status, priority.
- `data.conflicts` — массив конфликтов: task_id, conflict_with, work_center_id, overlap_start, overlap_end.

Поведение: приоритетная сортировка (URGENT → HIGH → NORMAL → LOW), учёт времени переналадки между продуктами, проверка пересечений задач на одном РЦ. В БД задачи не переводятся в IN_PROGRESS — только расчёт и возврат превью.

## Результаты на странице

После успешного «Запустить диспетчеризацию»:
- Таблица **Preview Gantt**: задача, рабочий центр, начало, окончание, длительность (ч), переналадка (мин).
- Блок **Конфликты в расписании** (если есть): какие задачи конфликтуют и на каком РЦ.

## Связь с другими разделами

- **Расписание** — заказы выпускаются в производство (release), создаются задачи; часть логики диспетчеризации есть и в ядре (`/api/v1/dispatching/*`).
- **DSIZ Planning** — планирование варок на горизонт; Shift Actualize — тактический шаг: «эта смена, эти заказы, превью и (позже) факт».
