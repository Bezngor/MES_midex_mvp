# Поведенческие правила пользователя при планировании производства

**Версия:** 1.0  
**Дата:** 02.02.2026  
**Статус:** Черновик для теста планирования и последующей формализации  

**Назначение:** Фиксация логики пользователя системы при работе с вкладкой «Стратегия» и пересчётом плана. Документ предназначен для последующего превращения в набор поведенческих правил (как их использовать — отдельная проработка).

---

## 1. Обновление списка заказов

**Правило:** Пользователь хочет обновить список заказов, созданных в 1С с момента последнего обновления.

**Действие в системе:** Кнопка «Обновить заказы покупателей» загружает новые и изменённые заказы (из БД или CSV). Пользователь видит списки «Новые заказы» и «Изменённые заказы».

---

## 2. Приоритеты

**Правило:** Пользователь видит заказы в разных приоритетах (Срочно, Высокий, Обычный, Низкий).

**Контекст:** Для пользователя критически важно выполнить в срок заказы с приоритетом **«Срочно»** — как правило, это самые важные заказы.

---

## 3. Поведение при принятии срочных заказов (URGENT)

Когда пользователь нажимает «Принять» для заказа «Срочно», система должна вставлять этот заказ в производственный процесс **так, чтобы он был выполнен накануне даты отгрузки**.

### 3.1. Просроченный срочный заказ

- Если заказ уже просрочен (не было сырья или тары):
  - При достаточности всех компонентов заказ ставится на **«завтра»** относительно дня планирования («сегодня»).
- Если недостаточно компонентов — поведение уточняется (ошибка резервирования, drill-down и т.д.).

### 3.2. Требуется варка массы

- Масса варится **как можно скорее**.
- Упаковка / фасовка / тубировка планируется на **следующий день после варки**.

### 3.3. Нет свободного места в смене

Если в день/смену, когда возможно производство, **нет свободного места** в расписании, система выбирает **первый слот, не разрывающий предыдущий заказ**.

**Пример (из описания пользователя):**  
В предполагаемый день уже есть задачи для РЦ:
- Операция1.2 — 2 ч (8:00–10:00) — вторая часть операции, начатой в предыдущую смену;
- Операция2 — 10 ч (10:00–20:00).

**Ожидаемое решение системы:**
1. Завершить начатую в прошлую смену **Операцию1.2** (8:00–10:00).
2. Вставить **СРОЧНАЯ1** — 4 ч (10:00–14:00).
3. Разделить **Операцию2** на две части:
   - Операция2.1 — 6 ч (14:00–20:00);
   - Операция2.2 — 4 ч (8:00–12:00 следующего дня).

То есть: срочный заказ вставляется **между** существующими операциями; уже начатые операции не разрываются; последующие сдвигаются (в том числе с переносом части на следующую смену).

### 3.4. Объединение с уже запланированным производством того же продукта

- Если в **ближайшие 2 дня** (от дня планирования) уже запланировано производство **того же продукта**, что и в срочном заказе, система должна:
  - Выбрать это запланированное производство и **добавить к нему количество** срочного заказа (объединение, без разрыва плана).
- Если такого варианта в ближайшие 2 дня **нет** — срочный заказ вставляется между существующими задачами; остальные сдвигаются (как в п. 3.3).

**Принцип:** Если есть возможность объединения заказов в ближайшие 2 дня, не разрывать сформированный план. Если возможности объединения нет — вставлять срочный заказ между существующими и сдвигать остальные.

---

## 4. Остальные заказы (не «Срочно»)

**Правило:** Заказы с приоритетом ниже «Срочно» **не имеют права разрывать** уже сформированные цепочки производственных заказов с приоритетом выше, чем у них.

**Примечание:** Детализацию и проверку этого пункта пользователь предполагает провести после теста срочных заказов.

## Связь с текущей реализацией

- Текущий алгоритм в `StrategicPlanningService` и `STRATEGIC_PLANNING_ALGORITHM.md` реализует базовый поток: резервирование BOM по приоритетам, планирование операций по маршруту с учётом смен и отсутствия пересечений.
- Описанные выше правила (вставка срочных без разрыва операций, объединение по продукту в 2 дня, запрет разрыва цепочек высшего приоритета) — целевое поведение для последующей доработки алгоритма и/или отдельного модуля правил.

## Типичные причины «Ошибок резервирования» при пересчёте плана

Если при нажатии «Пересчитать план» все заказы попадают в «Ошибки резервирования» (успешно зарезервировано: 0):

1. **Нет остатков на складе.** Резервирование BOM проверяет наличие компонентов (сырьё, масса, тара) в `inventory_balances`. После `prepare_test_env` остатки не загружаются — нужно отдельно загрузить остатки (см. DATA_LOAD_CSV.md, раздел «Загрузка остатков (Inventory)»).
2. **Нет BOM для продукта заказа.** В системе должен быть BOM для ГП (продукт заказа), и при необходимости BOM для Массы. Загружается из `dataset_bom.csv` при `prepare_test_env`.
3. **product_id заказа не совпадает с продуктом в BOM.** В заказах хранится UUID продукта (из справочника products). Если заказы или продукты загружались иначе, возможна рассинхронизация.

В интерфейсе после пересчёта с ошибками показывается блок «Ошибки резервирования» с перечнем заказов и указанием количества недостающих позиций (drill-down).
