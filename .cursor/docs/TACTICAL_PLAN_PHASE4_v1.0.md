# Тактический план Phase 4: Advanced Features (v1.2.0-dsiz)

## 1. Назначение документа

- **Утверждённый тактический план:** данный файл фиксирует согласованный с владельцем проекта план работ Phase 4.
- **Операционный инструмент:** используется как чек-лист и дорожная карта для итераций разработки, тестирования и доработок.
- **Перенос контекста:** является основным файлом переноса контекста между чатами/сессиями при исчерпании контекстного окна.
- **Единый источник правды:** состояние задач Phase 4 (что сделано, что в работе, что заблокировано) отражается именно здесь.

## 2. Регламент использования и обновления плана

- **2.1 Обновление статусов задач**
  - После каждой успешной итерации, корректировки кода или прохождения тестов
    статусы соответствующих задач/подзадач в секции `todos` и в тексте плана
    должны быть обновлены:
    - `pending` → задача ещё не начата;
    - `in_progress` → активно выполняется;
    - `completed` → реализована, протестирована, задокументирована;
    - `blocked` → временно заблокирована (указать причину в тексте).

- **2.2 Работа с предложениями по UI/UX**
  - Любые предложения пользователя по изменению интерфейса или поведения
    должны быть:
    - проанализированы на соответствие доменной логике и архитектурным правилам (см. `MES_RULES.md`, `PROJECT_MANIFEST_v1.1.md`);
    - если изменение **конфликтует** с основной логикой — **обязательно** запрашивается отдельное подтверждение пользователя
      с объяснением возможных последствий;
    - только после явного подтверждения такие изменения вносятся и фиксируются в этом плане.

- **2.3 Роль файла в переносе контекста**
  - При исчерпании контекстного окна новый диалог должен поднимать текущий контекст
    из следующих файлов:
    - `BUSINESS_PROCESS_REFERENCE.md` — **единый источник правды** по бизнес-процессу производства;
    - `CONTEXT_STACK_v1.1.md` — общий контекст проекта;
    - `PROJECT_MANIFEST_v1.1.md` — состояние проекта и фазы;
    - `TACTICAL_PLAN_PHASE4_v1.0.md` — детальный тактический план Phase 4 (этот файл).
  - В начале нового диалога необходимо кратко резюмировать:
    - текущие статусы ключевых задач из `todos`;
    - последние выполненные итерации и принятые решения;
    - соответствие реализуемых задач бизнес-процессу (`BUSINESS_PROCESS_REFERENCE.md`).

- **2.4 Изменения самого плана**
  - Любое изменение структуры плана (добавление/удаление разделов, смена приоритетов)
    должно:
    - быть согласовано в чате;
    - быть отражено как в тексте разделов, так и в секции `todos`;
    - по возможности сопровождаться краткой пометкой/датой изменения в тексте.

## 0. Реализация бизнес-процесса производства

### Единый источник правды: бизнес-процесс предприятия

**Важно:** Все доработки MES-приложения должны соответствовать бизнес-процессу производства, описанному в `.cursor/docs/BUSINESS_PROCESS_REFERENCE.md`. Бизнес-процесс является **единым источником правды**, а не текущая реализация MES.

**Принцип:** MES реализует бизнес-процесс предприятия, а не наоборот.

**Схема процесса:** `.cursor/_olds/info/MIDEX_orders.drawio.xml` (draw.io) / `.cursor/_olds/info/MIDEX_orders.drawio.pdf` (PDF)

### 0.7 Обязательный блок Phase 4: UI для стратегического и тактического планирования (ключевой алгоритм MES)

**Статус:** обязательное включение в план Phase 4.  
**Отчёт о тестировании:** `.cursor/docs/MANUAL_E2E_TEST_REPORT.md`.  
**Связь с бизнес-процессом:** `.cursor/docs/BUSINESS_PROCESS_REFERENCE.md`, разделы 3-4 (Вкладки "Стратегия" и "Тактика").

Ручное тестирование этапа 0.0 завершено. Выявлено: невозможность проследить в приложении полный жизненный цикл заказа (стратегическое планирование → создание заказа → тактическое планирование → выпуск в производство) из‑за отсутствия UI заказов и незавершённости UI DSIZ (Planning, Shift Actualize, Master Data — заглушки или зачаточное состояние).

**Причины выделения в отдельный обязательный блок:**

1. **Ключевой алгоритм MES для данного предприятия:** цепочка «стратегическое планирование (горизонт, нетто-потребность, варки) → тактическое планирование (смена, заказы, диспетчеризация) → выпуск в производство и фактизация» является ядром MES. Без полноценного UI эта цепочка не может быть ни протестирована сквозно, ни использована в эксплуатации.
2. **Разделение стратегии и тактики:** в текущем UI смешаны параметры на период и параметры на смену/день. Выделение блока закрепляет необходимость явного разделения и доработки обоих уровней планирования.
3. **Зависимость остальных доработок:** расширенная диспетчеризация, автоматическое создание партий по плану, drag‑and‑drop в Gantt и т.д. опираются на рабочий UI планирования и фактизации; без него они не дают законченного сценария для пользователя.
4. **Соответствие бизнес-процессу:** реализация этого блока закрывает шаги бизнес-процесса "Вкладка 'Стратегия'" и "Вкладка 'Тактика'" (см. `BUSINESS_PROCESS_REFERENCE.md`).

**Задачи бэклога, входящие в этот блок:**

- **Выявление новых/изменённых ЗП (0.8):** реализация шага "Блок обновления данных" → "Выявить новые и/или измененные ЗП" (см. задачу 0.8 выше).
- **Вкладка "Стратегия":**
  - Показать список ГП из новых ЗП (на основе результатов 0.8)
  - Показать список измененных ЗП с drill-down (на основе результатов 0.8)
  - Отметить ГП, которые принимаются к планированию/перерасчёту
  - Пересчёт планирования производства (уже есть, требуется интеграция)
  - Принять/Не принять результаты пересчёта
  - Сохранить результаты пересчёта производства
  - Принять/Не принять спланированный вариант
- **DSIZ Planning (вкладка целиком):** доработка модели отображения (смены/слоты/дни), исправление позиционирования Gantt, учёт приоритета MRP, состав персонала по сменам, режим маркировки, понятная визуализация плана.
- **Вкладка "Тактика":**
  - Выбрать день на Ганте
  - Скорректировать план на день
  - Сохранить результаты дневного плана
