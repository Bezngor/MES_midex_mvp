# План ручного тестирования MES (сквозной функционал)

**Назначение:** Чек-лист для ручного сквозного тестирования всего текущего функционала MES — от создания продукта до DSIZ Shift Actualize. Выполняется в параллельной ветке; по окончании предоставляется отчёт.

**Связь с тактическим планом:** Этап 0.0 в `TACTICAL_PLAN_PHASE4_v1.0.md`. Без прохождения этого этапа и получения отчёта дальнейшая разработка блоков 1+ приостановлена.

---

## 1. Продукты

- [ ] Открытие страницы «Продукты» — список загружается, нет белого экрана/ошибок.
- [ ] Создание продукта: название, тип (Сырьё / Масса / Упаковка / ГП), ед. изм. (по умолчанию «кг»), при необходимости поля для BULK. Код создаётся автоматически.
- [ ] Редактирование сохранённого продукта (кнопка «Редактировать») — форма открывается с данными, сохранение обновляет запись.
- [ ] Удаление продукта — модальное подтверждение, после подтверждения запись удаляется (или ошибка отображается корректно).
- [ ] Фильтр по типу продукта — отображение только выбранного типа.

## 2. BOM (спецификации)

- [ ] Открытие страницы «BOM» — загрузка продуктов и списка.
- [ ] Выбор категории продукта для редактирования спецификации: сначала «ГП» или «Масса», затем выбор продукта из списка (сырьё не в списке).
- [ ] Просмотр спецификации выбранного продукта — дерево компонентов или «Спецификация пуста».
- [ ] Добавление компонента в спецификацию:
  - [ ] Выбор категории компонента (Масса / Сырьё; ГП не выбирается).
  - [ ] Выбор продукта из выбранной категории.
  - [ ] Количество — пустое по умолчанию, ввод числа (запятая/точка), до 6 знаков после запятой.
  - [ ] Единица измерения подтягивается из справочника продукта.
  - [ ] Сохранение строки BOM.
- [ ] Удаление строки BOM — подтверждение, обновление списка.

## 3. Партии (Batches)

- [ ] Открытие страницы «Партии» — список батчей загружается.
- [ ] Фильтр по статусу (Все / PLANNED / IN_PROGRESS / COMPLETED).
- [ ] Создание батча: выбор продукта (масса), количество (кг). Успешное создание и появление в списке.
- [ ] В карточке батча отображается название продукта (не UUID).
- [ ] Кнопка «Запустить» для батча в статусе PLANNED — переход в IN_PROGRESS, фиксация времени старта (без ошибки «Not Found»).
- [ ] Кнопка «Завершить» для батча в статусе IN_PROGRESS — переход в COMPLETED, фиксация времени завершения.

## 6. MRP (Material Requirements Planning)

Проверка в последовательности: каждый следующий шаг опирается на результат предыдущего. Если шаги независимы — от простого к сложному.

### 6.1 Открытие страницы и консолидация заказов (база для всего MRP)

- [ ] **6.1.1** Открыть страницу «MRP» (`/mrp`). Страница загружается без белого экрана и критичных ошибок.
- [ ] **6.1.2** Автоматически вызывается консолидация заказов (горизонт по умолчанию 30 дней). Отображается индикатор «Консолидация заказов...», затем таблица.
- [ ] **6.1.3** В таблице консолидации есть колонки: **Продукт**, **Общее количество**, **Приоритет**, **Крайний срок**, **Источник заказов**. При наличии заказов (SHIP/IN_WORK) в БД — таблица не пуста; отображается «Найдено N продуктов для производства».

### 6.2 Проверка данных консолидации

- [ ] **6.2.1** Суммы по продуктам соответствуют ожиданию: «Общее количество» = сумма количеств по всем заказам этого продукта в пределах горизонта.
- [ ] **6.2.2** Приоритеты рассчитаны по срокам: URGENT (<7 дней), HIGH (7–14), NORMAL (14–30), LOW (>30). Крайний срок — самый ранний из заказов по продукту.
- [ ] **6.2.3** «Источник заказов» — число заказов (N заказ(ов)), совпадающее с количеством исходных заказов по этому продукту.
- [x] **6.2.4** Автоматическая сортировка по дате: в таблице сверху отображаются ГП с самой ранней датой (крайний срок по возрастанию).
- [x] **6.2.5** Кнопки фильтра по приоритету: «Все», URGENT, HIGH, NORMAL, LOW. При выборе приоритета отображаются только заказы с этим приоритетом; если заказов с приоритетом нет — кнопка неактивна (серая).

### 6.3 Горизонт планирования

- [ ] **6.3.1** Изменить «Горизонт планирования (дней)» (например, 7 или 60), нажать «Обновить». Список консолидации пересчитывается.
- [ ] **6.3.2** При уменьшении горизонта — меньше продуктов/строк (если заказы выходят за новый горизонт). При увеличении — не меньше, чем было (могут добавиться заказы).

### 6.4 Взрыв BOM (расчёт потребности в материалах)

Исходные данные: выбран один продукт из консолидации (ГП) и его количество. Шаг выполняется через API, если в UI нет кнопки «Взрыв BOM» для строки.

- [ ] **6.4.1** Вызвать `POST /api/v1/mrp/explode-bom` с `product_id` (UUID ГП из консолидации) и `quantity` (например, total_quantity по этому продукту). Ответ 200, в теле — `requirements` (словарь product_id → количество).
- [ ] **6.4.2** В `requirements` присутствуют компоненты спецификации: BULK (масса), при необходимости RAW, упаковка. Количества соответствуют BOM (например, доля массы × quantity ГП).
- [ ] **6.4.3** Для продукта без BOM или с пустой спецификацией — ожидаемая ошибка 400 или пустой `requirements` (по контракту API).

### 6.5 Нетто-потребность (валовая − доступный запас)

Исходные данные: один BULK (или другой компонент) из результата взрыва BOM и его валовая потребность.

- [ ] **6.5.1** Вызвать `POST /api/v1/mrp/net-requirement` с `product_id` (UUID BULK) и `gross_requirement` (кг из взрыва). Ответ 200: `net_requirement`, `available_stock`, `needs_production`.
- [ ] **6.5.2** Формула: `net_requirement = gross_requirement - available_stock` (но не меньше 0). При нулевых остатках `net_requirement` = валовая потребность, `needs_production` = true при net > 0.
- [ ] **6.5.3** При наличии остатков по этому продукту (страница «Остатки») — доступный запас учитывается, нетто уменьшается.

### 6.6 Округление до батча и создание заказа на варку

Исходные данные: BULK продукт с настройками `min_batch_size_kg`, `batch_size_step_kg` и нетто-потребность из шага 6.5.

- [ ] **6.6.1** Вызвать `POST /api/v1/mrp/create-bulk-order` с `bulk_product_id`, `quantity_kg` (нетто из 6.5), `due_date` (например, крайний срок из консолидации). Опционально `parent_order_id`. Ответ 200: создан заказ типа INTERNAL_BULK.
- [ ] **6.6.2** В ответе `rounded_quantity` ≥ `quantity_kg`, округление вверх до кратности `batch_size_step_kg` и не меньше `min_batch_size_kg`. Поле `original_quantity` = переданное `quantity_kg`.
- [ ] **6.6.3** В списке заказов (или на странице «Расписание» / заказы) созданный заказ отображается с новым номером, типом INTERNAL_BULK, количеством = rounded_quantity.
- [ ] **6.6.4** *(Отложено)* Вызов с `parent_order_id` (UUID существующего заказа на ГП) — вернуться к этому пункту при тестировании раздела «Заказы» / списка заказов, когда есть реальные заказы для подстановки в качестве родителя.

### 6.7 Граничные и ошибочные случаи

- [ ] **6.7.1** Пустой набор данных: нет заказов в БД в пределах горизонта — таблица консолидации пуста, сообщение «Нет заказов для консолидации», без падения.
- [ ] **6.7.2** Неверный `product_id` в explode-bom / net-requirement / create-bulk-order — ответ 400 или 404 с понятным сообщением.
- [ ] **6.7.3** Вызов net-requirement для продукта не типа BULK/RAW (например, ГП) — поведение по контракту (допустимо или ошибка); при необходимости проверить документацию API.

## 7. DSIZ Planning — **вся вкладка в бэклог после Dispatching**

Вся вкладка «DSIZ Planning» (форма параметров, запуск планирования, Gantt, сводка) перенесена в бэклог и будет дорабатываться/тестироваться **после** раздела Dispatching.

**Причины:** несоответствие модели отображения данным (12 рядов «реакторов» при 2 сменах × 2 слотах), смещение блоков Gantt относительно разбивки по дням, неочевидный алгоритм размещения операций, отложенные элементы (приоритет MRP, состав персонала, режим маркировки). Кнопка «Запустить планирование» только считает план в памяти; в БД ничего не создаётся.

**Бэклог (после Dispatching):**
- [ ] **7.бэклог** Полная доработка и тестирование вкладки DSIZ Planning: выравнивание модели (смены/слоты/дни), исправление позиционирования Gantt, приоритет MRP, персонал по сменам, режим маркировки, понятная визуализация плана.

## 8. DSIZ Shift Actualize

**Описание вкладки:** см. `.cursor/docs/DSIZ_SHIFT_ACTUALIZE_GUIDE.md`.

Вкладка «Фактизация смены» (`/dsiz/shift-actualize`): выбор даты и номера смены, ввод ID заказов (через запятую), кнопка «Запустить диспетчеризацию» — вызов `POST /api/v1/dsiz/dispatching/run` (preview расписания задач по РЦ, без записи в БД). Кнопки «Сохранить факт» и «Напечатать задание смены» — заглушки. Дата и номер смены в API не передаются, только список `manufacturing_order_ids`.

- [ ] Открытие страницы «DSIZ Shift Actualize» — загрузка данных.
- [ ] Ввод ID заказов, запуск диспетчеризации — превью (таблица задач, конфликты) отображается.
- [ ] Сохранение/отправка данных — кнопки «Сохранить факт» и «Напечатать задание» пока заглушки; при необходимости проверить сообщения.
