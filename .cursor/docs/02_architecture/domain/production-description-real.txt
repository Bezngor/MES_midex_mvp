# Техническое описание производственного процесса (Domain Description)
**Проект:** Автоматизация планирования производства ДСИЗ
**Версия:** 1.0 (MVP Scope)

## 1. Общие сведения и Временная модель

Предприятие производит дерматологические средства индивидуальной защиты (крема, пасты, гели, спреи, мыло). Процесс является дискретно-непрерывным: варка массы (Batch process) $\rightarrow$ фасовка (Discrete process).

### 1.1. График работы
*   **Режим:** 24/7.
*   **Смены:** 2 смены по 12 часов.
    *   Смена №1: 08:00 – 20:00.
    *   Смена №2: 20:00 – 08:00.

### 1.2. Квантование времени и Нормирование
В системе используются две модели расчета времени выполнения операций:

1.  **По физическому циклу (Варка/Реакторы):**
    *   Операция неделимая. Длительность фиксирована технологией (4–5 часов/цикл).
    *   *Ограничение:* Максимум 2 цикла на один реактор в одну смену.
2.  **По производительности (Фасовка/Маркировка):**
    *   Длительность рассчитывается динамически: $T = \frac{Volume}{Rate}$, где:
        *   $Volume$ — объем заказа (шт).
        *   $Rate$ — нормативная скорость (шт/час) для конкретного SKU на конкретном оборудовании.
    *   *Корректировка:* При ручных операциях $Rate$ зависит от количества персонала.

## 2. Топология производства и Ресурсы

### Секция 1: Приготовление массы (Bulk Production)
*Тип процесса: Периодический (Batch).*

Ключевое изменение: **Реактор физически один**. Он является «узким местом» (bottle-neck) для производства кремов и паст.

| Ресурс (Equipment) | Режим работы (Mode) | Производительность / Ограничения |
| :--- | :--- | :--- |
| **Универсальный Реактор**<br>*(Единственный физический агрегат)* | **Режим "Крем"** | -  **Загрузка:** Min 500 кг — Max 1000 кг.<br>-  **Цикл:** 4-5 часов.<br>-  **Макс. производительность:** до 2 варок в смену (всего до 2000 кг). |
| | **Режим "Паста"** | -  **Загрузка:** Min 750 кг — Max 1500 кг.<br>-  **Цикл:** 4-5 часов.<br>-  **Макс. производительность:** до 2 варок в смену (всего до 3000 кг). |
| **Миксеры (Кубы)**<br>*(Независимый ресурс)* | Гели, Мыло, Спреи | -  Работают **параллельно** с Реактором.<br>-  Макс. 4 куба одновременно в смену.<br>-  Механическое смешивание в сменных емкостях (IBC-контейнерах). |

**Ограничения планирования Секции 1:**
1.  **Взаимоисключение (Mutual Exclusion):** Нельзя одновременно варить Крем и Пасту. Реактор занимается только одной задачей в один момент времени.
2.  **Суммарная емкость смены:** Всего в смену (12 часов) можно выполнить не более **2-х циклов** варки (любой комбинации, например: 2 крема, ИЛИ 2 пасты, ИЛИ 1 крем + 1 паста).

### Секция 2: Тубировка (Tube Filling)
*Тип процесса: Автоматическая линия.*

| Линия | Продукт | Скорость (ориентир) | Особенности |
| :--- | :--- | :--- | :--- |
| **Линия 1** | Паста | ~9 000 шт/смена | -  Требует переналадки при смене массы.<br>-  Использует матрицу совместимости. |
| **Линия 2** | Крем, Гель | ~15 000 шт/смена | ^ То же самое |

### Секция 3: Полуавтоматический и Ручной розлив
*Тип процесса: Ручной/Полуавтоматический.*
*   **Продукция:** Малые серии, нестандартная тара (флаконы, канистры), срочные заказы.
*   **Ресурс:** Оборудование + **Люди**.
*   **Производительность:** Зависит от доступного персонала (шт/человеко-час).
*   **Особое требование:** Маркировка «Честный знак» (см. п. 2.1).

### Секция 4: Автоматический розлив жидкостей
*Тип процесса: Автоматическая линия.*
*   **Продукция:** Жидкое мыло, Спреи (флаконы 0.1–0.5 л).
*   **Производительность:**
    *   Мыло: ~5 000 шт/смена.
    *   Спрей: ~7 000 шт/смена.

### 2.1. Процесс Маркировки (Пост-обработка)
Продукция с Секций 3 и 4 требует обязательной маркировки «Честный знак».
*   **Ресурс:** Люди (ручной труд).
*   **Логика:** Обычно выполняется в смену $N+1$ после розлива, либо откладывается (накопление полуфабриката).
*   **Планирование:** Планируется как отдельная операция `OP_LABEL`, потребляющая человеко-часы.

## 3. Модель данных продукта (Product Data Model)

Система должна поддерживать многоуровневую спецификацию (BOM).

### 3.1. Уровни BOM (Иерархия)
1.  **Level 0 (Finished Good):** Готовая продукция (пример: *Паста "Северянин" 200мл*).
    *   Состоит из: `Масса` + `Туба` + `Гофроящик`.
2.  **Level 1 (Semi-Finished):**
    *   **Масса (Bulk):** Производится в Секции 1 (пример: *Масса Паста "Северянин"*). Состоит из сырья (Бентонит, ПАВы, Отдушки).
    *   **Туба:** Производится в Секции 5 или закупается. Состоит из Ламината и Гранул PP.
3.  **Level 2 (Raw Materials):** Сырье для варки и производства туб.

*Примечание для MVP:* Упаковка (гофроящики) в планировании пока не ограничивает процесс, но списывается по факту.

### 3.2. Матрица совместимости (Changeover Matrix)
Для минимизации времени переналадки на фасовочных линиях (Секции 2, 4) используется логика переходов.
Структура данных: `[Mass_A] -> [Mass_B] = [Status]`
*   **Совместим:** Переналадка минимальна (или 0 мин).
*   **Несовместим:** Требуется полная промывка (например, 60-90 мин).
*   **Универсальный:** Совместим со всеми (например, вода/спирт при промывке).
*   **Изолированный:** Всегда требует промывки до и после.

## 4. Логистические параметры и Ограничения

### 4.1. Человеческие ресурсы (Workforce Constraints)
*   Для ручных операций (Секция 3, Маркировка) ключевым ресурсом является **Фонд рабочего времени (ФРВ)**.
*   Входной параметр смены: `Количество людей`.
*   Возможность корректировки: Если 1 человек заболел, доступная мощность (Capacity) пересчитывается, длительность операций увеличивается.

### 4.2. Склад и Хранение (Inventory Limits)
1.  **Масса (Bulk Storage):**
    *   Хранилище: 2 стационарных накопителя (Паста) + Мобильные Кубы (IBC).
    *   **Лимит Кубов:** Физически на складе помещается **до 10 кубов**.
    *   Тара: Отсутствие *пустого* куба не является блокирующим фактором (их достаточно).
2.  **Незавершенное производство (WIP):**
    *   Разлитая, но не маркированная продукция может накапливаться в ожидании свободных людей.

### 4.3. Политика "Минимальной варки" (Under-loading Policy)
Если потребность в массе < Min загрузки реактора:
1.  **Этап 1 (MVP):** Планируется варка минимально допустимого объема (например, надо 100 кг $\rightarrow$ варим 500 кг).
    *   100 кг идут в заказ.
    *   400 кг становятся свободным остатком на складе (учитывается срок годности массы).
2.  **Этап 2 (Future):** Алгоритм ищет прогнозные заказы, чтобы утилизировать эти 400 кг сразу.

## 5. Алгоритм Планирования (Order Planning Logic)

Система выполняет планирование в 5 шагов:

### Шаг 1: Агрегация и Приоритеты
Сбор заказов со статусом `PLANNED`. Сортировка по приоритету:
1.  **Manual Urgent:** Ручные срочные заказы (наивысший приоритет, возможен override правил).
2.  **MTO (Make-to-Order):** Клиентские заказы с жестким дедлайном.
3.  **MTS (Make-to-Stock):** Пополнение до Min остатка.
4.  **Forecast:** Прогнозные заказы (заполнение мощностей).

### Шаг 2: Расчет Нетто-потребности (Net Requirements)
Для каждого SKU ГП:
$$Req_{Net} = \max(0, \text{Заказ} - \text{Свободный остаток ГП})$$
Результат округляется до кратности партии (если задана).

### Шаг 3: Взрыв Спецификации (BOM Explosion)
Расчет потребности в Массе:
1.  Определяем ID массы из BOM.
2.  Считаем требуемый объем (кг).
3.  Проверяем свободный остаток Массы на складе (включая остатки в кубах и накопителях).
4.  Формируем **Заказ на Варку** на недостающий объем.

### Шаг 4: Планирование Варки (Mixing Schedule) — *Обновлено*
Если сформированы заказы на варку (Batch Orders) для Крема и Пасты:

1.  **Очередь к Ресурсу (Resource Queue):** Все заказы на варку Крема и Пасты выстраиваются в единую очередь к ресурсу `MAIN_REACTOR`.
2.  **Разрешение коллизий:**
    *   Если необходимо сварить и Крем, и Пасту в одну смену, алгоритм занимает слоты последовательно (Слот 1: Крем $\rightarrow$ Слот 2: Паста).
    *   Если суммарная потребность превышает 2 цикла (например, нужно 3 варки), третий заказ принудительно переносится на следующую смену или на другой день.
3.  **Независимость Миксеров:** Заказы на массу для Гелей/Мыла/Спреев планируются на ресурс `MIXER_CUBES` и **не конкурируют** за время Реактора.

### Шаг 5: Планирование Фасовки (Filling Schedule)
1.  Операция фасовки `OP_FILL` может начаться не ранее, чем закончится `OP_MIX` (или сразу, если масса есть на остатках).
2.  Оптимизация последовательности (Sequence Optimization):
    *   Группировка заказов по типу массы для минимизации переналадок (используя матрицу совместимости).
3.  Расчет длительности: $Duration = \frac{Quantity}{Rate \times N_{pers}}$ (где $N_{pers}$ актуально для ручных линий).
4.  Проверка доступности людей (для Секции 3 и Маркировки).

## 6. Пример структуры данных (JSON Representation Concept)

Ниже приведена полная концептуальная структура данных (JSON Schema Concept), которая покрывает все сущности, ограничения и логику, обсужденную выше.

Эта структура разделена на три блока:
1.  **Master Data (Справочники):** Ресурсы, Продукты, Правила.
2.  **Input State (Входящее состояние):** Текущие запасы и Очередь заказов.
3.  **Output Plan (Результат планирования):** Сгенерированные операции.

### 3. Output Plan (Результат работы планировщика)

Это то, что система генерирует. Обратите внимание на связь `WARKA` -> `FASOVKA`.

```json
{
  "production_plan": {
    "generated_at": "2025-12-13T19:30:00",
    "operations": [
      {
        "id": "OP-MIX-001",
        "type": "MIXING",
        "resource_id": "RES_REACTOR_MAIN", // Занимаем Главный Реактор
        "output_product_id": "MASS-000772",
        "planned_quantity_kg": 750, // MIN загрузка (потребность была 10000 * 0.192 = 1920кг. Минус 1200 склад = 720кг. Округлили до min 750)
        "schedule": {
          "date": "2025-12-14",
          "shift_num": 1,
          "slot_index": 1, // Первый цикл в смене
          "estimated_start": "08:00",
          "estimated_end": "13:00"
        },
        "status": "SCHEDULED"
      },
      
      {
        "id": "OP-FILL-001",
        "type": "FILLING",
        "resource_id": "RES_LINE_TUBE_PASTE",
        "output_product_id": "GP-000952",
        "planned_quantity_pcs": 10000,
        "input_materials": [
          {
            "source_type": "INVENTORY", // Часть берем со склада
            "item_id": "MASS-000772",
            "quantity": 1200
          },
          {
            "source_type": "OPERATION", // Часть берем с варки OP-MIX-001
            "ref_op_id": "OP-MIX-001",
            "quantity": 720
          }
        ],
        "dependencies": ["OP-MIX-001"], // Нельзя начать, пока варка не даст продукт (или начнем параллельно, если технология позволяет)
        "schedule": {
          "date": "2025-12-14",
          "shift_num": 1, // Начинаем в ту же смену, но позже
          "estimated_start": "13:30", // Учтено время переналадки (30 мин)
          "estimated_end": "20:00" // Переход на след. смену, если не успеем
        }
      }
    ]
  }
}
```

### Ключевые моменты этой структуры для логики:

1.  **`concurrency_limit: 1` в Ресурсе Реактора:** Это флаг для алгоритма: "Если я поставил сюда задачу `OP-MIX-001` (Паста) с 08:00 до 13:00, я НЕ МОГУ поставить `OP-MIX-002` (Крем) в этот же слот".
2.  **`modes` в Реакторе:** Позволяет валидировать объем партии. Если пытаемся запланировать 600 кг Пасты — валидатор выдаст ошибку, так как для `PASTE_MODE` min = 750.
3.  **`source_type`: `OPERATION`:** Явно показывает прослеживаемость партии. Мы знаем, что этот тюбик наполнен массой именно с этой варки (важно для качества и учета).
