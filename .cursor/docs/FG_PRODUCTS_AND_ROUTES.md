# Список ГП в датасете и маршруты

Список получен **из БД** после загрузки датасета (продукты типа FINISHED_GOOD). Всего **41** ГП (три ГП с неуникальным кодом исключены из датасета). Коды в таблице ниже должны совпадать с тем, что показывает система (Расписание, алармы). Если в документе указан один код, а в системе — другой при том же наименовании, это признак **неуникального кода ГП** (ошибка 1С: один продукт должен иметь один код).

Чтобы вывести актуальный список снова:
```bash
python -m backend.src.db.list_fg_products
```
(из корня репозитория, с окружением backend; в Docker: см. DATA_LOAD_CSV.md.)

## Неуникальный код ГП (ошибка 1С)

Если один и тот же продукт (наименование) в источниках имеет **более одного кода** (например, в заказах — код A, в маршрутах/правилах — код B), это грубая ошибка на стороне 1С. Такие ГП:
- **исключаются** при загрузке датасета (load_dataset_from_csv выводит предупреждение и не создаёт для них продукты/заказы);
- **удалены** из тестовых датасетов (заказы, BOM, маршруты, правила);
- при конвертации xlsx→CSV строки с такими кодами не попадают в CSV.

Если в дальнейшем система обнаружит неуникальность кодов ГП (один и тот же наименование — разные коды в заказах), при загрузке возникнет аларм и эти ГП не будут загружены.

## Варианты заведения маршрутов

### Вариант А: Вы составляете маршруты (расширенный формат)

**Длительность операции — вычисляется.** В маршруте храним не «минуты на операцию», а **норму выпуска** (штук в смену или штук в смену на 1 человека). Длительность для Ганта/планирования считается по объёму заказа и норме.

**Формула (1 смена = 12 ч = 720 мин):**

- **AUTO_SHIFT / AUTO_SHIFT_CREW:** линия за смену даёт `capacity_pcs_per_shift` штук. Для заказа на `Q` штук:
  - **Длительность (мин)** = (Q / capacity_pcs_per_shift) × 720  
  - Пример: Тубировка 1 — 15 000 шт/смену. Заказ 10 000 крем Реген → 10 000/15 000 × 720 = **480 мин (8 ч)**. Заказ 5 000 крем Универсал → 5 000/15 000 × 720 = **240 мин (4 ч)**.

- **MANUAL_PER_PERSON:** один человек за смену наносит `capacity_pcs_per_shift_per_person` этикеток ЧЗ. В плане на человека `Q` штук:
  - **Длительность (мин)** = (Q / capacity_pcs_per_shift_per_person) × 720  
  - Пример: 5 000 этикеток ЧЗ/смену на человека, в плане 2 500 шт → 2 500/5 000 × 720 = **360 мин (6 ч)** на ЧЗ; остальные 6 ч — резерв под другую работу.

Итого: в маршруте задаём **нормы** (штук в смену / штук в смену на человека) и при необходимости длительность смены; **длительность операции** при планировании считаем по формуле от количества по заказу.

#### Уточнение 1. Типы нормирования

| Код в таблице | Описание | Как заполнять |
|---------------|----------|----------------|
| **AUTO_SHIFT** | Автоматический РЦ: норма — **штук в смену** (линия/принтер). | `capacity_pcs_per_shift` = макс. штук ГП в смену. Опционально: `crew_size`, `min_crew_to_operate`, `crew_productivity_rule` (см. Уточнение 1a). |
| **MANUAL_PER_PERSON** | Ручной/полуавтоматический РЦ: норма — **штук в смену на 1 человека**. | `capacity_pcs_per_shift_per_person` = штук в смену на одного работника. |
| **AUTO_SHIFT_CREW** | Линия розлива: выход зависит от **комплектности состава** (напр. 4 чел. — 100%, 3 чел. — 50%). | `capacity_pcs_per_shift` при полном составе, `crew_size` = полный состав (напр. 4), `crew_productivity_rule` — правило по составу (см. Уточнение 1a). |

В таблице/CSV для каждой **операции** (или для РЦ, если норма общая для РЦ) указываете:
- **norming_type**: `AUTO_SHIFT` | `MANUAL_PER_PERSON` | `AUTO_SHIFT_CREW`
- **capacity_pcs_per_shift** — для AUTO_* (штук ГП в смену)
- **capacity_pcs_per_shift_per_person** — для MANUAL_PER_PERSON
- **crew_size** — опционально (номинальный/полный состав для линий)
- **min_crew_to_operate** — опционально (минимальный состав для работы РЦ; см. Уточнение 1a)
- **crew_productivity_rule** — опционально (правило «N человек → X%»; см. Уточнение 1a)
- **crew_notes** — опционально, произвольный текст

#### Уточнение 1a. Условия работоспособности РЦ (crew_size, min_crew_to_operate, crew_productivity_rule)

От состава бригады зависит, может ли РЦ выполнить задание. Если условий не хватает — РЦ не может работать, а работники учитываются в **Резерве**. Эти правила можно задать в таблице маршрутов (или в справочнике РЦ).

| Поле | Описание |
|------|----------|
| **crew_size** | Номинальный/полный состав (человек). |
| **min_crew_to_operate** | Минимальное количество человек, при котором РЦ **может работать**. Если в смене меньше — РЦ не работает, все работники в Резерв. Если не указано, считаем: РЦ работает только при полном составе (min = crew_size). |
| **crew_productivity_rule** | Правило производительности по составу. Формат: пары «число человек = процент», через запятую, напр. `4=100,3=50,2=0`. 0% означает «не может работать» (все в Резерв). |

**Примеры по вашим РЦ:**

| РЦ | crew_size | min_crew_to_operate | crew_productivity_rule | Пояснение |
|----|-----------|----------------------|------------------------|-----------|
| Тубировка 1 / 2 | 2 | 2 | — | Оператор + упаковщик. Одного нет — РЦ не работает, оставшийся в Резерв. |
| Линия розлива 1 / 2 | 4 | 3 | 4=100,3=50,2=0 | 4 чел — 100%, 3 чел — 50%, 2 и менее — не работает, все в Резерв. |
| Линия производства тубы | 2 | 2 | — | 2 чел — работа; 1 и менее — не работает, все в Резерв. |

В CSV можно добавить столбцы **min_crew_to_operate** и **crew_productivity_rule** (текст, напр. `4=100,3=50,2=0`). При импорте их можно сохранять в БД/конфиг и использовать при планировании и диспетчеризации (проверка «может ли РЦ взять задание» и расчёт эффективной производительности).

#### Уточнение 2. Маркировка ЧЗ

| Код в таблице | Описание |
|---------------|----------|
| **AUTO_PRINTER** | Маркировка ЧЗ по умолчанию в процессе производства (принтер на автоматическом РЦ). Отдельной операции в маршруте нет. |
| **MANUAL_ONLY** | Только ручная маркировка ЧЗ (ручные/полуавтоматические РЦ). |
| **MANUAL_FALLBACK** | Обычно принтер; при сбое принтера добавляется операция ручного нанесения ЧЗ. |

В таблице/CSV для каждой **операции** (или для РЦ) указываете:
- **marking_type**: `AUTO_PRINTER` | `MANUAL_ONLY` | `MANUAL_FALLBACK`

**Важно: ручная маркировка ЧЗ при сбое принтера не блокирует линию.** При поломке принтера ЧЗ на автомат. линии (тубировка, розлив) ручная наклейка ЧЗ выполняется **вне этой линии** — на выделенной территории (**WC_CHZ_MANUAL_AREA**). Тубировка/линия розлива при этом **остаётся свободной** и может выпускать другую продукцию. На зоне ручной маркировки обычно работают сотрудники из Резерва; там одновременно могут наклеивать ЧЗ по нескольким видам ГП. Поэтому операцию «Маркировка ЧЗ (при сбое принтера)» в маршруте нужно указывать с **work_center_name = WC_CHZ_MANUAL_AREA**, а не с РЦ самой линии. В диспетчеризации задача по этой операции создаётся на WC_CHZ_MANUAL_AREA, линия (WC_TUBE_LINE_1 и т.д.) не занимается и доступна для других заказов. У РЦ WC_CHZ_MANUAL_AREA задают `parallel_capacity` > 1 (или достаточную «мощность»), чтобы учитывать одновременную работу по нескольким ГП.

Если для операции добавляется отдельная строка «Ручная маркировка ЧЗ» (при сбое принтера), задайте её отдельной операцией с `marking_type` = `MANUAL_ONLY`, **work_center_name** = **WC_CHZ_MANUAL_AREA** и пометкой в названии, что это fallback.

#### Формат таблицы/CSV для Варианта А (разделитель `;`)

Кодировка UTF-8. Заголовок — первая строка.

| Поле | Обязательное | Описание |
|------|--------------|----------|
| product_code | да | Код ГП (из списка выше). |
| operation_sequence | да | Порядок операции (1, 2, 3, …). |
| operation_name | да | Название операции (напр. «Сборка/Упаковка», «Маркировка ЧЗ ручная»). |
| work_center_name | да | РЦ из списка выше (напр. WC_REACTOR_MAIN, WC_TUBE_LINE_1, WC_FILL_LINE_1, WC_FILL_LINE_2). |
| norming_type | да* | AUTO_SHIFT \| MANUAL_PER_PERSON \| AUTO_SHIFT_CREW. *Обязателен для расчёта длительности по формуле. |
| capacity_pcs_per_shift | да* | Штук ГП в смену (для AUTO_*). *Нужен для расчёта длительности. |
| capacity_pcs_per_shift_per_person | да* | Штук в смену на 1 чел. (для MANUAL_PER_PERSON). *Нужен для расчёта длительности. |
| shift_minutes | нет | Длительность смены, мин (по умолчанию 720 = 12 ч). |
| estimated_duration_minutes | нет | Фиксированная длительность, мин — только fallback, если норма не задана (иначе длительность вычисляется). |
| crew_size | нет | Номинальный состав (для линий). |
| min_crew_to_operate | нет | Минимальный состав для работы РЦ (см. Уточнение 1a). |
| crew_productivity_rule | нет | Правило по составу, напр. 4=100,3=50,2=0 (см. Уточнение 1a). |
| crew_notes | нет | Произвольный текст. |
| marking_type | нет | AUTO_PRINTER \| MANUAL_ONLY \| MANUAL_FALLBACK. |

Пример строки CSV (длительность по заказу будет считаться из нормы). Операция 2 — на **WC_CHZ_MANUAL_AREA**, чтобы не блокировать тубировку:
```text
product_code;operation_sequence;operation_name;work_center_name;norming_type;capacity_pcs_per_shift;capacity_pcs_per_shift_per_person;shift_minutes;marking_type
00-000970;1;Сборка/Упаковка;WC_TUBE_LINE_1;AUTO_SHIFT;15000;;720;AUTO_PRINTER
```

#### Уточнение 4. ГП на нескольких РЦ и правила выбора

Один и тот же ГП может производиться на **нескольких** рабочих центрах. Например:

- **GECO Мыло жидкое, 250 мл, Флакон, Дозатор** может идти по умолчанию на **WC_FILL_LINE_1**, а при необходимости — на **WC_FILL_LINE_2** или на **WC_SEMI_AUTO_VISCOUS** (полуавтомат для вязкой массы).

**В таблице маршрутов** для этого ГП вы указываете **один** маршрут (типовой вариант), например одну операцию «Розлив/Упаковка» на **WC_FILL_LINE_1** с нормами. **Не нужно** дублировать три отдельных маршрута для трёх РЦ. Все **допустимые** РЦ (WC_FILL_LINE_1, WC_FILL_LINE_2, WC_SEMI_AUTO_VISCOUS) и **правило выбора** (количество > 1000 → автомат. линии, иначе полуавтомат) задаются в **отдельной таблице правил выбора РЦ** (см. Уточнение 5). Нормы для линии 2 и полуавтомата могут храниться там же (по РЦ) или в справочнике норм по операциям/РЦ.

**Правило выбора РЦ** (пример): если количество ГП **более 1000 шт** — направлять на автомат. линии (WC_FILL_LINE_1, WC_FILL_LINE_2); иначе — на полуавтомат (WC_SEMI_AUTO_VISCOUS). Такие правила не загружаются в таблицу маршрутов: они лежат в **отдельной таблице правил** (см. Уточнение 5).

#### Уточнение 5. Отдельная таблица правил и перенос в админку

**Отдельная таблица правил выбора РЦ** рекомендуется:

- **Таблица маршрутов** (manufacturing_routes, route_operations) хранит **операции** по шагам: какая операция, на каком РЦ, с какими нормами. Один маршрут может описывать «типовой» вариант (например, операция на WC_FILL_LINE_1).
- **Таблица правил выбора РЦ** (например, `product_routing_rules` или `product_work_center_routing_rules`) хранит: продукт, допустимые РЦ, приоритет, **условие** (напр. «количество > 1000» → автомат. линии, иначе полуавтомат). Так таблица маршрутов не перегружается логикой «если-то», а правила можно менять и дополнять независимо.

Пример структуры правил (идея для БД/админки):

| Поле | Описание |
|------|----------|
| product_id / product_code | ГП |
| work_center_name | РЦ (или work_center_id) |
| priority_order | Приоритет выбора (1 — предпочтительный) |
| min_quantity | Мин. объём заказа для этого РЦ (опционально) |
| max_quantity | Макс. объём для этого РЦ (опционально) |
| rule_expression | Текст правила или код условия (опционально, для сложных правил) |

**Перенос логики в админку:** да, это предусмотрено. Сущности (продукты, РЦ, маршруты, нормы, правила выбора РЦ) спроектированы как **нормализованные данные в БД**. В дальнейшем админка будет просто давать UI для редактирования тех же таблиц: маршруты, операции, нормы, правила выбора РЦ. Не придётся изобретать новый формат — те же таблицы и поля будут редактироваться через формы и списки в админке.

#### Как система определяет, какой маршрут и РЦ выбрать при выпуске заказа

**Исходная схема (без костылей):**

1. **В таблице маршрутов** для каждого ГП хранится **один** маршрут — типовой вариант. В нём указан **основной** РЦ (например, WC_FILL_LINE_1 для жидкого мыла, WC_FILL_LINE_2 для спрея). Остальные операции (например, маркировка ЧЗ на WC_CHZ_MANUAL_AREA) — как в маршруте.

2. **В таблице правил выбора РЦ** (product_routing_rules) для каждого ГП перечислены **все допустимые** РЦ для «подставляемой» операции и условия (приоритет, min_quantity, max_quantity). Пример: «количество ≥ 1001 → WC_FILL_LINE_1», «количество ≤ 1000 → WC_SEMI_AUTO_VISCOUS».

3. **При выпуске заказа в производство** система:
   - Берёт **единственный** маршрут продукта (типовой вариант).
   - По правилам product_routing_rules находит **первое подходящее** правило для данного заказа (по количеству: order_qty в пределах min_quantity..max_quantity, правила перебираются по приоритету). Получает **выбранный РЦ** (chosen_wc_id).
   - Для каждой **операции маршрута**: если операция выполняется на РЦ, который входит в список РЦ из правил для этого продукта («подставляемый» РЦ), то при создании задачи на эту операцию подставляется **выбранный по правилам РЦ** (chosen_wc_id). Остальные операции (например, ЧЗ) остаются на РЦ из маршрута.
   - В результате заказ получает задачи на нужном РЦ (например, при 21 шт. мыла — на Полуавтомате для вязкой массы, а не на Линии розлива 1), без дублирования маршрутов в данных.

**Итого:** маршрут один (основной РЦ в маршруте), правила задают распределение по количеству/условиям; при создании задач РЦ «подставляется» по правилам.
