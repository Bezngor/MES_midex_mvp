## 2. Customization Architecture (Архитектура кастомизации DSIZ)

### 2.1. Общий подход

- Базовое ядро (`backend/core/`) остаётся универсальным: модели, MRP, Dispatching, API, схемы не модифицируются напрямую. 
- DSIZ-специфика реализуется в `backend/customizations/dsiz/` через:
  - наследование от core-сервисов (MRP, Dispatching),
  - внедрение зависимостей через FastAPI DI (`dependency_overrides`), 
  - отдельные DSIZ-справочники и конфигурацию (reactor modes, workforce, changeover, labeling, QR-политики).
- Внешний API для DSIZ имеет отдельный префикс: `/api/v1/dsiz/...`.

Целевая структура каталога:

```text
backend/customizations/dsiz/
  ├── services/
  │   ├── dsiz_mrp_service.py
  │   ├── dsiz_dispatching_service.py
  │   ├── dsiz_labeling_service.py
  │   └── dsiz_workforce_service.py
  ├── models/
  │   ├── dsiz_master_data.py
  │   └── dsiz_changeover_matrix.py
  ├── schemas/
  │   ├── dsiz_planning_schemas.py
  │   └── dsiz_admin_schemas.py
  └── routes/
      ├── dsiz_planning_routes.py
      └── dsiz_admin_routes.py
```

### 2.2. DSIZMRPService (кастомизация MRP)

**Роль:** надстройка над core MRP с учётом DSIZ-ограничений. 

Основные обязанности:

- Расчёт нетто-потребности по ГП (как в core MRP), но с учётом: 
  - иерархии «ГП → Масса → Сырьё/Тара», 
  - того, что упаковка (гофроящики, тара) не ограничивает мощность и только списывается по факту. 
- Расчёт потребности в массе:
  - учёт текущих остатков Bulk (в том числе в кубах/накопителях), 
  - применение политики минимальной варки (округление до min загрузки реактора), 
  - генерация Batch Orders для реактора / кубов (отдельные ресурсы).
- Учёт календаря реактора:
  - лимит 2 варки/смену,
  - блокировка по понедельникам смена 1 (CIP). 

Интерфейс MRP API остаётся совместим с `MRP_GUIDE.md` и `API_SPEC.md`, DSIZ-поведение реализуется внутри сервиса.

### 2.3. DSIZDispatchingService (кастомизация диспетчеризации)

**Роль:** планирование очереди операций к ресурсам с учётом DSIZ-правил. 

Функции:

1) **Планирование варки:**

- Формирование единой очереди Batch Orders к `WC_REACTOR_MAIN` (крем + паста) с учётом:
  - взаимного исключения режимов,
  - лимита 2 цикла на смену,
  - CIP-блоков по понедельникам смена 1. 
- Планирование заказов на массу для гелей/мыла/спреев на `WC_MIXER_CUBES` (не конкурируют с реактором). 

2) **Планирование фасовки (OP_FILL):**

- Оптимизация последовательности по матрице совместимости (changeover matrix): 
  - группировка по `compatibility_group`,
  - добавление setup-времени (переналадка) к основному времени операции.
- Учитывание workforce-ограничений:
  - проверка, может ли конкретная линия работать в смену с данным составом людей (через DSIZWorkforceService),
  - пересчёт длительности операций при недоукомплектованности (например, авто-розлив при 3 операторах работает с 0.5 × производительность — параметр, который позже переедет в админку).

3) **Планирование маркировки (OP_LABEL):**

- При `labeling_mode = AUTO` для заказа:
  - маркировка считается встроенной в `OP_FILL`,
  - отдельные `OP_LABEL` не генерируются.
- При `labeling_mode = MANUAL`:
  - создаются операции `OP_LABEL` на людской ресурс (WC_LABELING),
  - по умолчанию планируются на смену N+1, но могут быть сдвинуты в смену N при наличии свободного ФРВ. 

4) **Учет ручных блокировок оборудования:**

- Начальник производства может вручную заблокировать любой Work Center (ремонт, внеочередная профилактика и т.п.).
- DSIZDispatchingService обязан:
  - **не пытаться планировать операции** на заблокированный ресурс,
  - **пересчитать план** с учётом этой блокировки (перенос операций на другие смены/ресурсы, если возможно),
  - учитывать ручные блокировки наравне с CIP и другими системными простоями.

5) **Учет доступности QR-кодов для маркировки:**

- По умолчанию предполагается, что все QR-коды ЧЗ для планируемых заказов уже доступны к моменту фасовки.
- Для отдельных заказов возможна ситуация «QR ещё не готов»:
  - в этом случае DSIZDispatchingService должен иметь доступ к дате готовности QR (вводится Начальником производства),
  - операции `OP_LABEL` для этого заказа не могут быть запланированы **раньше указанной даты**,
  - если QR недоступен, система:
    - либо откладывает только `OP_LABEL` (если допускается хранение полуфабриката без ЧЗ),
    - либо блокирует выпуск (опция будет зафиксирована в параметрах, далее вынесена в админку).
	
```python
def select_work_centers_for_fg(self, fg_sku: str, quantity: int) -> List[str]:
    routing = self.db.query("dsiz_product_work_center_routing")
    return sorted([
        wc.id for wc in routing[fg_sku]
        if routing[fg_sku][wc].is_allowed 
        and (routing[fg_sku][wc].min_quantity_for_wc is None 
             or quantity >= routing[fg_sku][wc].min_quantity_for_wc)
    ], key=lambda wc: routing[fg_sku][wc].priority_order)
```

### 2.4. DSIZLabelingService (маркировка и статусы продукции)

**Роль:** единая точка управления маркировкой ЧЗ и статусами «ПОЛУФАБРИКАТ → ГП». 

Функции:

- Управление DTO/схемами статусов:
  - UNLABELED_FG / WIP (полуфабрикат после фасовки),
  - FG (готовая продукция).
- Публичные методы (концепт):
  - `generate_labeling_operations(plan, qr_availability) -> list[OP_LABEL]` — генерация операций маркировки по плану фасовки и информации о доступности QR.
  - `recalculate_for_shift(shift_state, staff_state)` — пересчёт размещения `OP_LABEL` при изменении фактического состава людей.
- Взаимодействие с DSIZDispatchingService:
  - получает план фасовки и флаги AUTO/MANUAL,
  - накладывает ограничения по датам готовности QR-кодов,
  - формирует валидный набор `OP_LABEL`.

### 2.5. DSIZWorkforceService (ограничения по персоналу)

**Роль:** инкапсулировать всю бизнес-логику по людям и их влиянию на мощности. 

Функции:

- Хранение норм укомплектованности по Work Center:
  - Реактор: 1 оператор (иначе невозможна варка).
  - Тубировка: 2 человека (оператор + упаковщик) — оба обязательны; без любого линия стоит, второй уходит в резерв.
  - Авто-розлив: 4 оператора, допускается работа с 3 с коэффициентом снижения производительности (например, 0.5; численное значение — параметр).
  - Производство туб (на будущее): 2 оператора, оба обязательны.
  - Остальное оборудование: гибкие правила (без строгих норм).
- Методы:
  - `can_run(work_center_id, staff_state) -> bool` — можно ли запускать станок в текущую смену.
  - `get_effective_rate(work_center_id, staff_state, base_rate) -> effective_rate` — пересчёт производительности по факту людей.
- Учет состояния «РЕЗЕРВ»:
  - сотрудники, не задействованные на критичных участках, считаются резервом,
  - часть участков допускает пополнение из резерва (авторозлив), часть — нет (тубировка, линия туб).

DSIZWorkforceService используется внутри DSIZDispatchingService и DSIZLabelingService при расчёте длительности операций и проверки возможности их выполнения.

### 2.6. DSIZ Master Data & Changeover (справочники)

**Роль:** хранить DSIZ-специфичные конфигурации, которые в MVP захардкожены, но в обязательном порядке будут вынесены в админку.

Список ключевых справочников:

- **Ресурсы и режимы:**
  - reactor modes (CREAM_MODE, PASTE_MODE, min/max загрузка, max_cycles_per_shift, CIP-календарь), 
  - nominal_rate_per_shift/час для линий тубировки, авто-розлива и ручного участка. 
- **Changeover matrix:**
  - пары `compatibility_group_from → compatibility_group_to → статус (COMPATIBLE/INCOMPATIBLE/...) + setup_time_min`. 
- **Workforce rules:**
  - нормальные численности по станкам и коэффициенты деградации производительности.
- **Labeling / QR-правила:**
  - флаги AUTO/MANUAL по умолчанию,
  - политика поведения при отсутствии QR (только откладываем OP_LABEL или блокируем выпуск полностью).
- **Rate directory:**
  - базовые скорости SKU × оборудование для расчёта длительности `OP_FILL` и `OP_LABEL`.

Архитектурно:

- На уровне MVP: Python-константы/конфигурационные файлы (например YAML), подключаемые в DSIZ-сервисах.
- В дальнейшем (обязательно): перенос в БД (отдельные таблицы) и админку, с возможностью:
  - единичного редактирования,
  - пакетной загрузки (Excel/CSV),
  - версионирования настроек.

### 2.7. DSIZ Planning API (`/api/v1/dsiz/...`)

Отдельное пространство API для DSIZ-логики:

**Планирование:**

- `POST /api/v1/dsiz/planning/run`
  - Запускает полный цикл: MRP (DSIZMRPService) + Dispatching (DSIZDispatchingService) + Labeling (DSIZLabelingService).
- `GET /api/v1/dsiz/planning/plan`
  - Возвращает текущий DSIZ-план: список операций `OP_MIX`, `OP_FILL`, `OP_LABEL` с привязкой к сменам, ресурсам, людям и QR-ограничениям.

**Админка DSIZ (параметры и справочники):**

- `GET/PUT /api/v1/dsiz/master-data` — ресурсы, режимы, reactor settings.
- `GET/PUT /api/v1/dsiz/changeover-matrix` — матрица совместимости.
- `GET/PUT /api/v1/dsiz/rates` — базовые производственные скорости.
- `GET/PUT /api/v1/dsiz/workforce-rules` — нормы по людям и коэффициенты.
- `GET/PUT /api/v1/dsiz/labeling-rules` — дефолтный режим ЧЗ, политика при отсутствии QR.
- `GET/PUT /api/v1/dsiz/qr-availability` — ввод/обновление дат готовности QR по заказам.
